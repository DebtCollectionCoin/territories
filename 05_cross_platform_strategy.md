# Cross-Platform Strategy & Porting Plan

> **Principle:** Build the engine once in pure Kotlin. Port only the UI layer per platform.
> The `:engine` module must remain platform-agnostic at all times.

---

## Platform Rollout Timeline

```
Phase 1 — Android (primary)
  → Build, test, polish, release on Google Play

Phase 2 — Windows Desktop
  → Kotlin Multiplatform (KMP) + Compose for Desktop

Phase 3 — Web
  → Kotlin/JS + Compose for Web (or React wrapper)

Phase 4 — Linux
  → Same binary as Windows (Compose for Desktop ships universal JVM jar)

Phase 5 — Apple (iOS + macOS)
  → KMP + Compose Multiplatform for iOS
```

---

## Architecture for Multi-Platform

```
territories/
├── engine/              ← Pure Kotlin (JVM/KMP) — shared everywhere
├── app-android/         ← Android-specific UI + DI + persistence
├── app-desktop/         ← Windows + Linux (Compose for Desktop)
├── app-web/             ← Browser (Kotlin/JS)
└── app-ios/             ← iOS (KMP + Swift interop)
```

### The Golden Rule
Any code that compiles only for one platform must live in that platform's module.
The `engine/` module imports nothing platform-specific.

## Phase 2 — Windows Desktop

**Technology:** [Compose Multiplatform](https://www.jetbrains.com/lp/compose-multiplatform/) (JetBrains)

### What's Shared with Android
- Entire `engine/` module (100%)
- All domain use cases
- Game logic and AI

### What Needs to Be Written
- `main()` entry point (Compose for Desktop `application {}`)
- Desktop window setup (title, icon, resizable, min size)
- Keyboard shortcuts: Ctrl+Z (undo), Esc (menu), etc.
- Native file dialogs (save/load game as JSON file)
- Persistence: SQLite via exposed or SQLDelight (no Room on desktop)

### Tasks

- [ ] **D-01:** Add `app-desktop` module; configure Compose Multiplatform
- [ ] **D-02:** Move shared UI composables into `shared-ui` module (BoardCanvas, HUD, etc.)
- [ ] **D-03:** Desktop `main.kt` with `application { Window(...) { ... } }`
- [ ] **D-04:** Adapt BoardCanvas touch → mouse click handling
- [ ] **D-05:** Add keyboard shortcut handler
- [ ] **D-06:** Replace Room with SQLDelight or a JSON file-based save system
- [ ] **D-07:** Package as `.exe` installer (Conveyor or jpackage)
- [ ] **D-08:** Code sign for Windows SmartScreen bypass

### Desktop-Specific UX
- Larger board visible without zooming (bigger default window = 60×40 grid fits)
- Right-click → context menu (undo, surrender)
- Hover highlights the nearest intersection before clicking
- Window title shows current score: "Territories — A: 12 | B: 8"

## Phase 3 — Web (Browser)

**Technology:** Kotlin/JS + Compose for Web (Canvas renderer) **or** compile engine to WASM/JS and wrap in React.

### Option A: Kotlin/JS + Compose Web (Recommended)
- Full Kotlin stack, engine reused directly
- Compose for Web is in active development — verify stability before committing
- Persistence: `localStorage` or IndexedDB

### Option B: Engine → WebAssembly (Fallback)
- Compile `engine/` to WASM via Kotlin/WASM target
- Write UI in TypeScript/React
- Bridge: JS calls into WASM functions
- Use if Compose Web is not stable enough

### Tasks

- [ ] **W-01:** Evaluate Compose for Web stability; decide Option A vs B
- [ ] **W-02:** Add `app-web` module; configure Kotlin/JS or WASM target
- [ ] **W-03:** Implement canvas rendering in JS context
- [ ] **W-04:** Implement touch + mouse input
- [ ] **W-05:** URL-based game state (share a game via URL)
- [ ] **W-06:** Deploy to static hosting (Cloudflare Pages / GitHub Pages / Netlify)
- [ ] **W-07:** PWA manifest (installable from browser, offline capable)
- [ ] **W-08:** Responsive layout: works on mobile browser too

## Shared UI Module Strategy

To maximize code reuse, extract platform-agnostic Compose composables into a shared module:

```
shared-ui/
├── BoardCanvas.kt       ← Platform-agnostic Compose Canvas rendering
├── GameHud.kt           ← Score display, controls (abstract callbacks)
├── HomeScreen.kt        ← Home screen layout
├── ResultScreen.kt      ← Results layout
└── theme/
    ├── Color.kt
    ├── Theme.kt
    └── Type.kt
```

Each platform module provides:
- Navigation host
- Persistence adapter
- Platform-specific input handling
- Platform entry point (`MainActivity`, `main()`, `index.html`, etc.)

## CI/CD Pipeline (Recommended)

```yaml
# .github/workflows/build.yml

jobs:
  test-engine:
    # Run all engine unit tests on every push
    runs-on: ubuntu-latest
    steps:
      - ./gradlew :engine:test

  build-android:
    runs-on: ubuntu-latest
    steps:
      - ./gradlew :app-android:assembleRelease

  build-desktop:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - ./gradlew :app-desktop:packageDistributionForCurrentOS

  build-ios:
    runs-on: macos-latest  # Xcode required
    steps:
      - ./gradlew :app-ios:iosSimulatorArm64Test
```

## Versioning Strategy

| Platform | Distribution | Versioning |
|----------|-------------|------------|
| Android | Google Play | `versionCode` (auto-increment in CI) |
| Windows | Direct download + winget | Semantic version in installer |
| Linux | `.deb` / `.rpm` / Snap | Semantic version |
| Web | Auto-deploy on push to `main` | Git SHA in footer |
| iOS | App Store | Build number (auto-increment) |

All platforms share the same `version.properties` file at the root.
